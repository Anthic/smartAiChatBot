### вњ… PHASE 2 вЂ” Vector Store Setup (Day 1, ~1 hour) (done)

**Goal:** Qdrant running and writable.

**Decision:** Run Qdrant locally using in-memory mode (no Docker needed for MVP). Switch to Docker for production.

### вњ… PHASE 3 вЂ” RAG Ingestion Pipeline (Day 2, ~2 hours) (done)

**Goal:** Upload file в†’ chunks stored in Qdrant.

1. Write `app/rag/embeddings.py`:
   - Wraps Mistral's `mistral-embed` model
   - Accepts a list of text strings
   - Returns a list of 1024-dim float vectors
   - Batch properly (Mistral embed max: 512 tokens per chunk recommended)

2. Write `app/rag/ingestion.py`:
   - `load_document(file_path, file_type)` в†’ raw text string
     - PDF: use `pymupdf` (`fitz.open()`)
     - TXT: plain `open().read()`
     - MD: use `markdown` to strip to plain text
   - `chunk_text(text)` в†’ list of string chunks
     - Use `RecursiveCharacterTextSplitter` with `chunk_size=512`, `chunk_overlap=64`
     - **Document your chunk size decision in README** (evaluators look for this)
   - `ingest_document(file_path, file_type)` в†’ calls load в†’ chunk в†’ embed в†’ upsert
     - Each chunk stored with metadata: `{source_file, chunk_index, text}`

3. Write `app/api/routes/ingest.py`:
   - `POST /ingest` вЂ” accepts `multipart/form-data` file upload
   - Saves file temporarily to `/tmp/`
   - Calls `ingest_document()`
   - Returns `{chunks_stored: N, status: "success"}`

**Exit Criteria:** Upload a PDF в†’ check Qdrant has N vectors stored.

### вњ… PHASE 4 вЂ” RAG Retrieval (Day 2, ~1 hour) (done)

**Goal:** Query в†’ get relevant chunks back.

1. Write `app/rag/retriever.py`:
   - `retrieve(query_text, top_k=3)` function
   - Embeds the query using `embeddings.py`
   - Calls `qdrant_client.search_vectors(query_vector, top_k)`
   - Returns list of `{text, score, source_file}` dicts
   - If best score < `0.65` threshold в†’ return empty list (no relevant context found)
   - **Document your score threshold decision in README**

**Exit Criteria:** Query a question about the uploaded doc в†’ get relevant chunks.

### вњ… PHASE 5 вЂ” Memory System (Day 2, ~45 min) (done)

**Goal:** Conversation context persists within a session.

1. Write `app/agent/memory.py`:
   - `ConversationMemory` class backed by a `collections.deque(maxlen=10)` (last 10 turns)
   - Methods: `add_turn(role, content)`, `get_history()` в†’ list of `{role, content}` dicts
   - `clear()` method
   - A `MemoryStore` dict-based registry: `{session_id: ConversationMemory}`
   - `get_or_create_session(session_id)` в†’ returns the right memory object
2. Memory must be passed to the LLM as `messages` array in every call
3. Keep it **in-process for now** (no Redis) вЂ” document this limitation in README

**Exit Criteria:** Two-turn conversation: "My name is John" в†’ "What's my name?" в†’ correct answer.

### вњ… PHASE 7 вЂ” The Agent Router (Day 3, ~2 hours вЂ” MOST IMPORTANT)

**Goal:** The brain that decides what to do with each message.

1. Write `app/agent/prompts.py`:
   - `SYSTEM_PROMPT` вЂ” tells Mistral it is an AI assistant for Studio Butterfly
   - Includes instructions: when to use tools, when to say fallback message
   - Include the exact fallback string: `"I couldn't find that information in the uploaded documents."`
   - Keep it as a constant вЂ” easier to document decisions

2. Write `app/agent/router.py` вЂ” the `AgentRouter` class:

   **The routing algorithm (explain every step in README):**

   ```
   Step 1: Load conversation history from memory
   Step 2: Call Mistral with tool_definitions attached
   Step 3: Check the response finish_reason:
       - "tool_calls"  в†’ extract tool name + args в†’ execute_tool() в†’ feed result back в†’ get final response
       - "stop"        в†’ check if response is useful
           - If response text contains fallback keywords в†’ attempt RAG retrieval
               - If RAG returns chunks в†’ rebuild prompt with context в†’ re-call Mistral
               - If RAG returns nothing в†’ return the fallback sentence as-is
           - If response looks fine в†’ return it directly
   Step 4: Save both user message and assistant reply to memory
   Step 5: Return final response text
   ```

   > This Mistral-native approach is cleaner than LangChain: Mistral decides tools, you decide RAG fallback.

**Exit Criteria:** "Where is my order ORD001?" в†’ tool called, correct answer returned.

## рџ”‘ Key Senior Dev Principles Applied

1. **No full LangChain** вЂ” only `langchain-text-splitters`. This gives you a clean, explainable architecture the evaluator can read in 5 minutes.
2. **Mistral native tool calling** вЂ” don't try to regex-parse tool intent. Let Mistral's `finish_reason: "tool_calls"` do it properly.
3. **Qdrant over FAISS** вЂ” production databases beat file-based indexes. Shows you're thinking ahead.
4. **Central `prompts.py`** вЂ” all prompts in one place. This is what separates a junior project from a senior one.
5. **`registry.py` pattern** вЂ” tools are pluggable, not hardcoded in the router.
6. **Document every decision** вЂ” the README IS the evaluation. Treat it as a first-class deliverable.
